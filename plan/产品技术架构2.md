### 一、 核心架构的极客级解剖 (Deep Dive Architecture)

在这套“便携式”架构中，虽然所有东西都在用户的电脑上运行，但我们在逻辑上把它做成了**“微服务架构”**。Python 是客户端，`llama-server` 是服务端。

#### 1. 进程通信机制 (IPC)
* **启动阶段：** Python 的主程序（UI）启动时，使用 `subprocess.Popen` 悄悄在后台拉起 `llama-server.exe`。
* **通信协议：** Python 不直接调用 C++ 代码，而是通过标准的 **HTTP REST API** 与 `llama-server` 通信（走本地环回地址 `http://127.0.0.1:8080`）。
* **为什么这么设计？** 极其稳定！就算 AI 模型因为遇到了奇怪的指令卡死了，后台的 `llama-server` 崩溃了，你的前端 Python UI 也不会闪退。Python 只需要捕获一个“请求超时”，然后重启一下后台进程即可。

#### 2. “暗度陈仓”的数据读取策略 (DuckDB + Excel 桥接)
* DuckDB 读 CSV 文件天下无敌，但原生读 Excel (`.xlsx`) 需要依赖外部插件，这对于“便携版”打包不友好。
* **我们的神仙解法：**
    1. 用户传入 `数据.xlsx`。
    2. Python 在后台用 `pandas` 极速提取它的第一行（表头），并将整个 Excel 静默转换为一个隐藏的 `temp_data.csv`。
    3. 把这个 CSV 喂给 DuckDB，DuckDB 用极低的内存瞬间完成 SQL 的分组、计算。
    4. DuckDB 输出结果 CSV，Python 再把它转换回 `处理结果.xlsx` 给用户。

### 二、 Python 源码级的模块设计 (Module Blueprint)

你的 `DataCopilot.py` 核心代码需要分为以下 **4 个独立的 Class（类）**，各司其职：

#### 模块一：`LLM_Engine` (大模型引擎控制器)
这个模块是管家，负责大模型的生杀大权。
* **核心依赖库：** `subprocess`, `requests`, `time`, `atexit`
* **关键方法：**
    * `start()`: 核心启动命令。你需要写死这些参数来保住你的 8GB 内存：
        `llama-server.exe -m model.gguf -c 1024 --threads 4 --batch-size 128 --port 8080`
        *(解释：`-c 1024` 把记忆卡死在 1024 个 Token，防止内存溢出；`--threads 4` 只用你 i5 处理器的 4 个核心，留一半给系统，防止鼠标卡顿。)*
    * `stop()`: 绑定在程序的退出事件上，确保用户点右上角红叉时，强制 `kill` 掉后台进程，绝不留内存垃圾。

#### 模块二：`Excel_DB_Bridge` (数据桥接器)
负责提取上下文，并执行 SQL。
* **核心依赖库：** `pandas`, `duckdb`
* **关键方法：**
    * `get_schema(file_path)`: 只读取 `nrows=1`。返回列名列表，例如 `["员工编号", "姓名", "薪资", "部门"]`。这步内存消耗不到 5MB。
    * `execute_sql(sql_query)`:
        建立 DuckDB 内存连接：`conn = duckdb.connect(':memory:')`
        执行查询：`conn.execute(sql_query).df()` (将结果转为 DataFrame 保存)。

#### 模块三：`Copilot_Brain` (业务大脑与 Prompt 组装)
这是整个产品**智商最高**的地方。负责和 LLM 聊天、纠错。
* **关键方法：`chat_and_run(user_input, schema)`**
    1.  **拼接 Prompt：**
        *“你是一个精通 DuckDB SQL 的数据分析专家。当前数据表名为 `temp_table`，包含列：{schema}。用户需求：{user_input}。请只输出一句可以立即执行的 SQL。如果需求模糊或缺少关键列，请输出反问句向用户澄清。禁止输出任何 Markdown 格式的解释代码！”*
    2.  **发送请求：** 向 `127.0.0.1:8080/v1/chat/completions` 发送 JSON 请求。强制设定 `temperature=0.1`（让模型保持极度理智，不发散瞎聊）。
    3.  **自我纠错循环 (While 循环)：**
        * 尝试用 `Excel_DB_Bridge` 执行返回的 SQL。
        * 如果捕获到 `duckdb.BinderException` (比如模型把“薪资”错写成了“工资”)，把错误信息 `Error: column "工资" not found` 塞回给 LLM，告诉它：“执行报错了，根据表头重新生成 SQL”。
        * 最大重试次数：2 次。如果 2 次都错，向用户报错：“指令太复杂，请换个说法”。

#### 模块四：`UI_Window` (极简交互层)
* **核心依赖库：** `tkinter`, `threading`
* **关键设计：** **绝对不能在主线程调 LLM！** 必须用 `threading.Thread` 开一个子线程去跑上面的 `Copilot_Brain`，否则在模型思考的那 3-5 秒里，你的软件界面会显示“无响应”，用户体验极差。

### 三、 目录结构的打包演变 (From Dev to Production)

为了让你清楚自己要写哪些文件，以及最后怎么变成那个神奇的 `.exe`，看这个对比：

**开发阶段（你的工作区）：**
```text
My_Project/
 ├── main.py                (把上面四个模块组合在一起的主入口)
 ├── logic_brain.py         (存放 Copilot_Brain 和 Excel_DB_Bridge)
 ├── server_manager.py      (存放 LLM_Engine)
 ├── llama-server.exe       (预编译的引擎)
 └── qwen2.5-1.5b.gguf      (模型文件)
```

**发布阶段（执行 PyInstaller 打包）：**
你在终端输入类似这样的一条终极命令（我会教你怎么写配置文件）：
`pyinstaller --noconsole --add-data "llama-server.exe;engine" main.py`

**打包后的最终形态（发给用户的样子）：**
```text
DataCopilot_V1/
 ├── _internal/             (一堆被压扁的 Python 库、DLL 库、和 llama-server.exe)
 ├── DataCopilot.exe        (唯一的主程序，带一个好看的 Icon)
 └── models/
     └── qwen2.5-1.5b.gguf  (由于模型太大，一般不直接打包进 exe，而是放在旁边让 exe 去读取)
```

### 四、 这套方案的绝对优势总结

通过这种颗粒度极细的设计，我们实现了：
1.  **物理级防爆内存：** 通过 `llama-server` 的启动参数和 DuckDB 的轻量化，死死按住了 8GB 的命门。
2.  **AI 的幻觉克星：** 通过“温度控制(0.1)” + “SQL 语法死板要求” + “报错自动重试 2 次”机制，把 1.5B 模型的胡说八道概率降到了最低。
3.  **对小白绝对友好：** 没有控制台黑框，不用安装任何依赖环境。
