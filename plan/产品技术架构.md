### 二、 软件内部解剖图 (Internal Architecture)

我们将软件拆分为 3 个高度解耦的微模块，这三个模块将在你的 `main.py` 中独立运作：

#### 1. 进程守卫模块 (Process Guardian)
* **真实机制：** 使用 Python 原生的 `subprocess.Popen`。
* **启动命令（严格限制内存）：**
  ```bash
  # 这是在 Python 中真实拼接执行的命令
  llama-server.exe -m models/qwen2.5.gguf --port 8080 --threads 4 -c 1024 --batch-size 128
  ```
  *(参数解释：`--threads 4` 限制只用 4 核 CPU 防卡顿；`-c 1024` 强制把上下文记忆卡在 1024 tokens，确保模型内存永远锁死在 1.5GB 左右。)*
* **退出机制：** 使用 Python 原生 `atexit.register(kill_process)`，确保即使软件崩溃，也能强杀 `llama-server.exe` 进程。

#### 2. “内存直连” 数据管道 (Data Pipeline)
* **步骤 A (加载数据)：** 当用户选择文件时，执行 `df = pandas.read_excel("用户数据.xlsx")`。此时数据存在内存里。
* **步骤 B (提取表头)：** 执行 `schema = df.columns.tolist()`，拿到类似 `['部门', '姓名', '销售额']` 的列表。
* **步骤 C (执行 SQL)：** 当 AI 返回 SQL（例如 `SELECT 部门, SUM(销售额) FROM df GROUP BY 部门`）时，直接调用 `result_df = duckdb.query(sql_string).df()`。
* **步骤 D (导出结果)：** 执行 `result_df.to_excel("处理结果.xlsx")`。
* **可行性：** DuckDB 与 Pandas 底层共享 Apache Arrow 内存格式，查询速度是毫秒级，且**不产生内存翻倍**。

#### 3. AI 会话与纠错引擎 (LLM Router & Self-Correction)
* **真实机制：** 使用 `requests` 库向 `http://127.0.0.1:8080/v1/chat/completions` 发送请求。
* **Prompt 约束策略（防幻觉核心）：**
  系统提示词必须写得极其死板：“表名为 `df`。列名有：[列名]。你只能输出纯 SQL 语句，绝不能包含 Markdown 的 ` 
* **捕获与重试循环 (The Retry Loop)：**
  ```python
  # 真实可行的伪代码逻辑
  for attempt in range(2): # 最多重试2次
      sql = get_llm_response()
      if sql.startswith("CLARIFY:"):
          return "UI 显示反问"
      try:
          duckdb.query(sql) # 尝试执行
          break # 成功则跳出循环
      except duckdb.Error as e:
          # 把报错信息丢回给大模型
          append_to_chat_history(f"SQL执行错误: {str(e)}。请修正 SQL 并重新输出。")
  ```

### 四、 这个架构为什么在 Surface Pro 8 上绝对不会翻车？

1. **规避了 Python 处理数据的短板：** 如果用 Pandas 强写多步数据处理，极容易因为中间变量没释放导致内存溢出（OOM）。DuckDB 是流式查询引擎，自带内存溢出保护。
2. **剥离了重型框架：** 没有使用 LangChain，没有用 LlamaIndex。纯手写 `requests` 调用和提示词拼接，把框架带来的几百兆内存开销彻底清零。
3. **物理隔离：** Python (管 UI 和 数据) 和 C++ (管 AI 推理) 在两个独立的进程里跑，谁也不会把谁卡死。
