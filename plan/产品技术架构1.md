### 一、 整体技术架构设计 (Technical Architecture)

系统在运行时，主要分为三个独立且互不干扰的层级。这种设计的好处是：**AI 推理的巨大计算压力被隔离在一个单独的进程中，绝对不会导致你的前端 UI 卡死。**

#### 1. 表现与控制层 (UI & Control Layer)
* **技术栈：** Python + `Tkinter` (原生 UI) + `subprocess` (进程管理)
* **职责：**
    * 提供极简的用户界面（文件选择按钮、对话输入框、运行日志输出）。
    * 作为“大管家”，负责在启动时悄悄拉起底层的 `llama-server.exe`，并在用户关闭软件时，精准“杀死”这个后台进程，释放内存。

#### 2. 核心逻辑与数据层 (Logic & Data Layer)
* **技术栈：** Python + `DuckDB` + `Pandas` (仅用于辅助读写)
* **职责：**
    * **上下文感知：** 极速读取 Excel 的表头信息（列名和数据类型），不加载全量数据，内存消耗几乎为零。
    * **Prompt 组装：** 将“表头信息 + 用户自然语言指令 + 强制澄清/SQL生成规则”拼接成标准的系统提示词。
    * **SQL 执行与回退：** 接收 AI 返回的 SQL 语句，交给 DuckDB 在内存中执行。如果 DuckDB 报错，捕获错误信息并触发重试机制。

#### 3. 本地 AI 服务层 (Local AI Service Layer)
* **技术栈：** `llama-server.exe` (C++ 编译的可执行文件) + `Qwen2.5-1.5B-GGUF` 模型
* **职责：**
    * 作为一个纯本地的 HTTP 服务器运行（默认监听 `http://127.0.0.1:8080`）。
    * 接收控制层发来的 Prompt，进行纯 CPU 推理（利用 i5-1135G7），并返回生成的 SQL 语句或澄清提问。

### 二、 软件目录结构 (Software Directory Structure)

当你的项目开发完毕，经过 PyInstaller 打包后，最终呈现给用户的**“一站式绿色免安装包”**长这样。总大小约 1.3GB - 1.5GB。

```text
DataCopilot_v1.0_Portable/  (用户解压后的主文件夹)
│
├── _internal/              (存放 AI 引擎和模型的隐藏/核心目录)
│   ├── engine/
│   │   └── llama-server.exe     # 核心发动机：预编译的 llama.cpp 服务端 (约 25MB)
│   │
│   └── model/
│       └── qwen2.5-coder-1.5b-instruct-q4.gguf  # 核心大脑：1.5B 量化模型 (约 1.1GB)
│
├── DataCopilot.exe         # 用户唯一需要双击的主程序 (包含 Python 逻辑和 UI，约 60MB)
│
└── 使用说明.txt             # 简单的文本提示（说明：双击 exe 即可，无需联网）
```
*注：`DataCopilot.exe` 在运行时，会通过相对路径自动去 `_internal` 文件夹里寻找引擎和模型。*

### 三、 Python 核心模块拆解 (Module Breakdown)

在你的 Python 源码（即将被打包成 `DataCopilot.exe` 的代码）中，需要划分以下四个核心代码模块：

| 模块名称 | 核心类/函数 | 具体职责 | 关键防崩设计 (针对 8GB 内存) |
| :--- | :--- | :--- | :--- |
| **1. 进程管理器** | `ModelManager` | 负责 `subprocess.Popen` 启动 `llama-server`。 | 启动命令中**必须强制附加** `-c 1024` (限制上下文长度) 和 `-t 4` (限制使用 4 个 CPU 核心)，防止榨干 Surface 资源。 |
| **2. 表格解析器** | `ExcelSchemaReader` | 读取 Excel 第一行，生成列名列表。 | 绝对不要用 `pandas.read_excel()` 读取整个文件！只需读取 `nrows=1` 或使用轻量级的 `openpyxl` 只读模式。 |
| **3. API 通讯官** | `LLMClient` | 使用 `requests` 库向 `localhost:8080/v1/chat/completions` 发送请求。 | 设置 `timeout=60`，因为 1.5B 纯 CPU 推理需要时间；强制设定 `temperature=0.1`，确保 AI 输出 SQL 的稳定性，不乱发散。 |
| **4. 引擎执行器** | `DuckDBExecutor` | 执行 SQL 语句：`duckdb.query("SELECT ... FROM 'xxx.xlsx'")` | 采用 `try...except` 结构，如果捕捉到 SQL 语法错误，立刻将错误字符串丢回给 `LLMClient` 要求重写。 |

### 四、 运行时数据流 (Data Flow)

为了让你更直观地理解代码的运行过程，我们模拟一次用户输入：*“把销售额大于5000的挑出来”*。

1.  **[UI 触发]**：用户点击“发送”按钮。
2.  **[提取表头]**：Python 瞬间读取文件，提取出 `[姓名, 部门, 销售额]`。
3.  **[构建 Prompt]**：Python 将其拼接为：
    > "你是一个精通 DuckDB SQL 的 AI。当前有一张表，包含列：[姓名, 部门, 销售额]。用户需求：‘把销售额大于5000的挑出来’。请只输出 SQL 语句，禁止输出其他废话。如果是模糊指令，请提问。"
4.  **[本地网络请求]**：Python 将上述 Prompt 作为 JSON，发送给后台安静运行的 `llama-server.exe`。
5.  **[CPU 推理]**：Surface 的 i5 处理器开始狂飙（约 3-5 秒）。
6.  **[返回结果]**：`llama-server` 返回：`SELECT * FROM excel_data WHERE 销售额 > 5000`。
7.  **[执行导出]**：Python 拦截到这条 SQL，传入 DuckDB。DuckDB 在内存中瞬间处理完毕，并触发 `to_csv()` 或 `to_excel()` 导出新文件。
8.  **[UI 响应]**：界面提示“处理成功，已在桌面生成新表格”。
